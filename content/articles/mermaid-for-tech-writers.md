Диаграммы — неотъемлемая часть технической документации. Они помогают читателю быстрее понять архитектуру системы, последовательность вызовов или структуру данных. Но традиционный подход "нарисовал в draw.io → экспортировал в PNG → вставил в статью" создаёт проблемы: картинки устаревают, их сложно версионировать, а обновление одного узла требует перегенерации всего изображения.

Mermaid решает эту проблему, превращая диаграммы в код. В этой статье разберу, почему для технического писателя Mermaid удобнее PlantUML, какие типы диаграмм наиболее полезны, и почему текстовый формат диаграмм идеально подходит для работы с ИИ-агентами.

## Mermaid vs PlantUML: зачем менять привычный инструмент

PlantUML поддерживается почти везде, имеет огромное количество типов диаграмм и зрелую экосистему. Но у него есть фундаментальный недостаток: он требует Java-рантайма или серверной рендерилки. В современном docs-as-code подходе это создаёт трение.

| Критерий | Mermaid | PlantUML |
|---|---|---|
| Рендеринг | Нативно в браузере (JavaScript) | Требует Java или Graphviz |
| GitHub / GitLab / Notion | Нативная поддержка из коробки | Нет или только через плагины |
| Интеграция в SSG | Простая: один JS-файл | Сложнее: нужен сервер или CLI |
| Синтаксис | Компактный, читаемый | Подробный, иногда многословный |
| Редактирование агентами | Легко: текстовый формат | Легко, но синтаксис длиннее |

{% mermaid() %}
flowchart TD
    A[Sync Process] --> B{Conflict Detected?}
    B -->|No| C[Apply Changes]
    B -->|Yes| D{Strategy}
    D -->|Manual| E[Queue for Review]
    D -->|Source Wins| F[Use Source Data]
    D -->|Target Wins| G[Use Target Data]
    D -->|Newest Wins| H[Compare Timestamps]
    E --> I[Manual Resolution]
    F --> C
    G --> C
    H --> C
    I --> C
{% end %}

Mermaid выигрывает там, где важна скорость итераций и низкий порог входа. Вы пишете диаграмму прямо в Markdown, и она отображается в GitHub, GitLab, Notion, Obsidian и большинстве статических генераторов сайтов без дополнительной инфраструктуры.

## Архитектура системы: flowchart

Flowchart — самый универсальный тип диаграмм в Mermaid. Он подходит для описания архитектурных схем, пользовательских сценариев и процессов. Ниже — пример архитектуры сервиса синхронизации данных.

{% mermaid() %}
graph LR
    A[Data Source] -->|Read| B[PinkCloudSync API]
    B -->|Write| C[Target System]
    C -->|Reverse Sync| B
    B -->|Update| A
    B -->|Detect Conflicts| D[Resolution Module]
    D -->|Apply Strategy| B
{% end %}

Для сложных схем с множеством уровней и подграфов полезен ELK-layout. Ниже — расширенная версия той же архитектуры с группировкой компонентов. ELK распределяет узлы более упорядоченно, что особенно заметно на больших диаграммах.

{% mermaid(layout="elk") %}
flowchart TD
    subgraph Sources
        A[PostgreSQL]
        B[REST API]
        C[CSV Files]
    end

    subgraph Processing
        D[PinkCloudSync API]
        E[Sync Engine]
        F[Conflict Resolver]
    end

    subgraph Targets
        G[MySQL]
        H[MongoDB]
        I[Webhook]
    end

    A --> D
    B --> D
    C --> D
    D --> E
    E --> F
    E --> G
    E --> H
    E --> I
    F --> D
{% end %}

## Sequence diagram: взаимодействие сервисов

Sequence diagram незаменим, когда нужно показать порядок вызовов между компонентами системы. Ниже — типичный сценарий аутентификации: сначала клиент получает токен, затем использует его для API-запросов.

{% mermaid() %}
sequenceDiagram
    participant Client
    participant API

    Client->>API: POST /auth/token<br/>(api_key, api_secret)
    API->>Client: access_token (JWT)

    Client->>API: GET /sync/sessions<br/>(Authorization: Bearer token)
    API->>Client: Response data
{% end %}

Sequence diagram хорошо масштабируется: можно добавлять `autonumber` для нумерации сообщений, `rect` для выделения фрагментов и `note over` для комментариев. Всё это читается прямо в исходном Markdown.

## State diagram: жизненный цикл объекта

State diagram описывает переходы между состояниями. Это удобно для моделирования workflow публикации статей, CI/CD-пайплайнов или обработки заказов. Ниже — жизненный цикл сессии синхронизации данных.

{% mermaid() %}
stateDiagram-v2
    [*] --> pending: Create session
    pending --> in_progress: Start sync
    in_progress --> completed: Success
    in_progress --> failed: Error
    failed --> [*]
    completed --> [*]
{% end %}

Каждый переход помечен событием, которое его вызывает. Такая диаграмма сразу отвечает на вопрос: "Что может произойти дальше?"

## Mermaid и ИИ-агенты: почему текстовый формат побеждает

Главное преимущество Mermaid для работы с LLM — диаграмма представлена обычным текстом. Это означает три вещи:

1. **Создание.** Агенту не нужно генерировать бинарные файлы или координаты SVG. Достаточно выдать структурированный текст в синтаксисе Mermaid.
2. **Обновление.** Изменение одного узла или связи — это правка одной строки. Diff читается человеком и машиной одинаково легко.
3. **Извлечение информации.** Диаграмму можно распарсить регулярными выражениями или простым скриптом, чтобы извлечь список компонентов, зависимостей или состояний.

{% mermaid() %}
graph LR
    A[Create Session] --> B[Data Reading]
    B --> C[Processing]
    C --> D{Conflicts?}
    D -->|Yes| E[Resolve]
    D -->|No| F[Write Data]
    E --> F
    F --> G[Complete]
{% end %}

Сравните с графическими редакторами: там агент должен был бы оперировать координатами пикселей или JSON-схемой draw.io, что на порядок сложнее и хуже поддаётся контролю.

## Стилизация и адаптация под дизайн

Mermaid предлагает несколько встроенных тем: `default`, `dark`, `forest`, `neutral` и `base`. Последняя позволяет полностью переопределить цвета через `themeVariables`.

Кроме того, Mermaid рендерит диаграммы в SVG. Это значит, что они масштабируются без потери качества на любых экранах, а через CSS можно дополнительно стилизовать контейнер или фон.

На этом блоге реализовано автоматическое переключение темы диаграмм при смене светлой/тёмной темы сайта. Это достигается за счёт повторной инициализации Mermaid с нужной темой при переключении режима.

## Дополнительные возможности, о которых стоит знать

- **Кликабельные узлы.** В flowchart можно сделать любой узел ссылкой: `click A href "/api-reference"`. Это полезно для интерактивных архитектурных схем.
- **GitHub-native.** Mermaid рендерится прямо в README, issues и pull requests GitHub без плагинов.
- **Accessibility.** SVG-вывод можно снабдить заголовками и описаниями, что делает диаграммы доступнее для скринридеров.
- **Экосистема.** Расширения для VS Code, Obsidian, Notion и Confluence делают Mermaid частью привычного рабочего процесса.

## Чеклист: какую диаграмму выбрать

| Задача | Рекомендуемый тип |
|---|---|
| Показать архитектуру или алгоритм | Flowchart |
| Сложная схема с подграфами | Flowchart + ELK layout |
| Последовательность вызовов API | Sequence diagram |
| Жизненный цикл, workflow | State diagram |
| Структура данных или API | Class diagram |
| Пропорции, доли | Pie chart |
| Динамика во времени | XY chart |

## Заключение

Mermaid — это не просто замена PlantUML, а инструмент, который лучше вписывается в современный docs-as-code подход. Он работает в браузере, интегрируется в любой SSG, рендерится на GitHub и идеально подходит для генерации и редактирования ИИ-агентами.

Если вы ещё не пробовали Mermaid, начните с [Mermaid Live Editor](https://mermaid.live/) — он позволяет экспериментировать с синтаксисом в реальном времени без установки чего-либо.
