В 2025–2026 годах выбор языковой модели для рабочих задач напоминает шоппинг в гипермаркете без ценников: ассортимент огромен, маркетинг кричит обо всем сразу, а понять, что действительно подойдет под конкретную задачу, можно только методом проб.

## Практический фреймворк выбора

1. **Определите класс моделей** — облачные API (frontier) или локальное развёртывание (on-prem)
2. **Выберите категорию по активным параметрам** — ≤7B, 7–20B, 20–50B, 50B+/frontier
3. **Подберите конкретные модели внутри категории** — сравните active params, контекстное окно, языковую поддержку
4. **Проверьте на реальных задачах** — создайте вайб-сюиту из 5–10 типовых промптов

Начинайте с самых мощных frontier-моделей через API, оцените выгоду, а затем ищите локальные альтернативы для оптимизации стоимости.

## Две большие категории задач

Вся работа с языковыми моделями в области документирования сводится к двум категориям: **вспомогательные задачи** и **создание контента**. Различие между ними определяет требования к модели сильнее, чем любые цифры в спецификации.

### Вспомогательные задачи

Это всё, что помогает поддерживать базу знаний в рабочем состоянии, но не порождает новый материал для читателя напрямую: speech-to-text, построение knowledge graphs, форматирование, ревью, мониторинг свежести, поиск по корпусу. Для поиска языковая модель генерации не нужна — используйте эмбеддер (multilingual-e5-large или bge-m3) и гибридный поиск: семантический через векторы плюс BM25/TF-IDF. Чисто векторный поиск пропускает точные совпадения по терминологии.

Для этих задач критичны **надёжность, скорость и предсказуемость**. Модель должна уметь следовать инструкциям, формировать структурированный вывод и не уходить в творческую импровизацию.

### Создание контента

Здесь модель генерирует текст, который напрямую читает аудитория. Но и эта категория раскалывается на два подтипа.

**Работа в условиях высокой неопределенности** — задачи, где нет единого источника истины, а результат зависит от синтеза разрозненных данных: архитектурная документация из бесед, обучающие материалы для разных аудиторий, PRD из десяти тикетов. Для таких задач нужны **frontier-модели** (GPT-4o, Claude 4, Gemini 2.5).

**Работа в условиях низкой неопределенности** — задачи с чёткими входными данными и заранее определённым форматом: release notes из git, changelog, переводы, API-справка из OpenAPI. Здесь нужна **аккуратность и низкая латентность**, а не топовый интеллект.

**Пример: документирование из записи встречи.** Три задачи подряд: транскрибация (любая модель), суммаризация (минимум 20B параметров, лучше 35B+), натягивание на шаблоны (для простых документов хватит 70B, для сложных — frontier). Проверял на своём пет-проекте — ассистенте для личной базы знаний. Qwen 3.6 35B дешевле GPT-OSS 120B на 30%, но по качеству не дотянул. При том что я — единственный пользователь и не очень требовательный.

## Почему мультимодальность обязательна

Современная техническая документация — это не только текст, но и скриншоты, диаграммы, архитектурные схемы, скринкасты. Мультимодальная модель умеет:

1. **Описать изображение** — превратить скриншот UI в текстовое описание для документации
2. **Извлечь информацию из схем** — прочитать архитектурную диаграмму и сгенерировать объяснение
3. **Обработать видео** — суммаризировать скринкаст или запись демонстрации
4. **Создать иллюстрации по описанию** — набросать схему по текстовой спецификации

Без мультимодальности вы теряете 30–50% контекста технической документации. Локальные мультимодальные варианты: Qwen-VL, DeepSeek-VL, Yi-VL. В облаке — frontier-модели через API.

## Ландшафт моделей и выбор

**Frontier-модели** (GPT-4o, Claude 4, Gemini 2.5) способны на всё, но дороги и требуют отправлять данные на внешние серверы. **Китайские модели** (DeepSeek-V3, MiniMax, Qwen 3) доступны через OpenRouter или напрямую и дают 80–90% качества frontier за существенно меньшие деньги.

*Примечание для российских реалий 2026:* доступ к frontier-моделям через API может быть ограничен, а требования 152-ФЗ к локализации данных делают локальное развёртывание или использование российских провайдеров (GigaChat, YandexGPT) юридически безопасным выбором. Для русскоязычных задач обращайте внимание на поддержку русского в токенизаторе — Qwen и DeepSeek показывают лучшие результаты. Также учитывайте экспортные ограничения на оборудование — GPU A100/H100 могут быть недоступны, но есть альтернативы: Huawei Ascend, Baidu Kunlun или использование CPU-инференса с квантованными моделями.

**On-prem open-source.** До 2024 года стандартом были dense-модели. С 2025-го open-weight модели среднего и крупного размера почти исключительно **MoE** (Mixture-of-Experts). Dense сохранили relevance только в малом формате (≤14B). Для всего остального рынок предлагает MoE: DeepSeek-V3, MiniMax-Text-01, Qwen 3 MoE, Llama 4, GPT-OSS 120B.

При выборе on-prem модели смотрите на **активные параметры** — они определяют вычислительную сложность и потолок рассуждений. Общее число параметров влияет на широту знаний, но не переносит модель в более высокий класс по способностям.

| Модель | Всего / активные | Что это значит |
|---|---|---|
| Qwen 3.6 35B-A3B | 35B / 3B | Работает на MacBook; скорость как у dense 7B |
| GPT-OSS 120B | 117B / 5.1B | Помещается на одном GPU, но инференс — как у 7B |
| Llama 4 Maverick | 400B / 17B | Требует 4+ GPU для загрузки; потолок — 17B active |
| DeepSeek-V3 | 671B / 37B | Требует 2× A100; лидер по цене/качеству |
| MiniMax-Text-01 | 456B / 46B | Мощная китайская MoE; требует серьёзного железа |

При локальном развёртывании измеряйте три метрики на вашем железе: **TTFT** (время до первого токена — критично для интерактива), **Latency** (общее время ответа) и **TPS** (токенов в секунду). MoE с 3B active на MacBook выдаёт 40–60 токенов/с; MoE с 37B active на A100 — 15–25 токенов/с. Но высокий TPS у мелких active params не компенсирует ограниченный потолок рассуждений.

### На что обращать внимание при выборе

- **Активные параметры × контекстное окно** — главный предиктор способностей, а не total params.
- **Эффективное контекстное окно** — «128K» на коробке часто означает 32K–64K до деградации. Проверяйте needle-in-haystack.
- **Упрямство инструкций** — мелкие active params упрямее (хорошо для JSON/YAML), большие кооперативнее (хорошо для мозгового штурма).
- **Квантование** — тестируйте конкретную комбинацию total/active params и квантизации. «Запускается» ≠ «работает».
- **Токенизатор** — Qwen (~4 символа/токен) вмещает в «128K» на ~30% больше текста, чем Llama 2 (~2.5 символа/токен).

Оценивайте модель комплексно, а не по одному бенчмарку. Синтетические тесты вроде MMLU полезны только для отсева заведомо сломанных моделей. Для русскоязычных задач смотрите Russian SuperGLUE — но помните, что бенчмарки измеряют общие способности, а не качество на ваших конкретных промптах. Агентские бенчмарки (SWE-bench, TAU-bench) ближе к реальности, но единственная истина — ваша собственная «вайб-сюита» из 5–10 реальных задач. Прогоняйте одинаковые промпты при температуре 0.3–0.6 и оценивайте: accuracy, format adherence, recovery, latency.

Стресс-тесты находят точку поломки: needle-in-haystack (обычно деградация после ~60% контекста), repetition trap (обработка 50+ элементов), adversarial formatting (грязный Markdown и OCR-испорченный текст).

### Чего ожидать от каждого класса

| Класс (active params) | Сильные стороны | Слабости | Лучший юзкейс | Примеры |
|---|---|---|---|---|
| ≤7B | Быстрая, работает на MacBook | Слабое многоходовое рассуждение, забывает инструкции | Форматирование, классификация, edge | Qwen 3.6 35B-A3B (3B), GPT-OSS 120B (5B) |
| 7–20B | Локальный sweet spot, надёжный JSON/YAML | Борется с graduate-математикой, теряет нить в >10 шагов | KB-агент, простой контент | Llama 4 Scout/Maverick (17B) |
| 20–50B | Сильный синтез, надёжное использование инструментов | Требует 2× A100 или квантование | Сложный анализ, создание контента | DeepSeek-V3 (37B), MiniMax-Text-01 (46B) |
| 50B+ / Frontier | Лучшие рассуждения, устойчивость к неопределенности | Дорого, данные уходят наружу | Высокая неопределенность | GPT-4o, Claude 4, Gemini 2.5 |

Класс active params — это фильтр по железу и способностям одновременно. MoE с 3B active может обогнать 17B active на конкретном цикле KB-агента благодаря лучшему контекстному окну или обучению инструментам. Но 17B active дадут качественно более глубокий синтез.

## Итоговая матрица

| Задача | Рекомендуемый диапазон active params | Примеры моделей (всего / active) |
|---|---|---|
| Speech-to-text | Whisper base → Whisper large-v3 | Не LLM |
| Форматирование, опечатки | ≤7B active | Qwen 3.6 35B-A3B (35B / 3B) |
| Поиск по корпусу | Эмбеддер + reranker | bge-m3 (568M), e5-large (335M) |
| Ревью документации | ≤7B active → 7–20B active | Qwen 3.6 35B-A3B → Llama 4 Scout (109B / 17B) |
| Release notes | ≤7B active → 7–20B active | GPT-OSS 120B (117B / 5B) → Llama 4 Scout |
| Перевод документации | ≤7B active → 7–20B active | Qwen 3.6 35B-A3B → Llama 4 Scout |
| Построение knowledge graphs | ≤7B active → 7–20B active | GPT-OSS 120B → Llama 4 Scout |
| Создание обучающих материалов | 20–50B active → Frontier API | DeepSeek-V3 (671B / 37B) → GPT-4o / Claude 4 |
| Архитектурная документация из бесед | 20–50B active → Frontier API | MiniMax-Text-01 (456B / 46B) → GPT-4o / Claude 4 |
| Мониторинг свежести документов | ≤7B active | Qwen 3.6 35B-A3B |

Главное правило: **начинайте с ваших задач как с бенчмарка**, используйте синтетические оценки только для отсева кандидатов, и всегда тестируйте на длине контекста и квантизации из продакшена.

## Экспериментальная секция: реальные промпты

Вот несколько реальных промптов, которые я использую для тестирования моделей на моих задачах:

**Промпт для суммаризации встречи (русский):**
```
Из транскрипции встречи извлеки ключевые решения, назначенные задачи и сроки. Ответ в формате JSON:
{
  "decisions": ["список решений"],
  "tasks": [{"who": "имя", "what": "задача", "deadline": "срок"}],
  "next_meeting": "дата следующей встречи или null"
}
```

**Промпт для создания API-документации:**
```
На основе OpenAPI-спецификации ниже создай документацию для разработчиков:
1. Краткое описание каждого endpoint
2. Примеры запросов и ответов
3. Возможные ошибки и их решения
4. Best practices для использования
```

**Промпт для перевода технического текста:**
```
Переведи следующий технический текст с английского на русский, сохраняя:
1. Терминологию (не переводи названия технологий)
2. Форматирование кода и команд
3. Техническую точность
```

Эти промпты я прогоняю через 3–5 кандидатов с температурой 0.3 и сравниваю: accuracy, format adherence, latency, cost.

## Чек-лист для быстрого выбора

1. **Задачи**: вспомогательные (≤7B active) или создание контента (7–50B+ active)?
2. **Бюджет**: облачный API (frontier дорого) или локальное развёртывание (железо + время)?
3. **Данные**: можно отправлять наружу (API) или только локально (152-ФЗ)?
4. **Язык**: нужна поддержка русского (Qwen/DeepSeek) или достаточно английского?
5. **Контекст**: какая длина документов (≤64K, 64–128K, 128K+)?

Ответы на эти вопросы сузят выбор до 2–3 моделей, которые стоит протестировать на вашей вайб-сюите.
